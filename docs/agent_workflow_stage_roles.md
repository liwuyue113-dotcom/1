# Agent Workflow 阶段作用说明

## 文档用途

本文用于解释 Tool Calling / Memory / FastAPI / MCP 这一组阶段分别解决什么问题。它主要服务于作品集录屏、答辩和面试讲解，不是新的功能规格。

总体一句话：

```text
这一组阶段证明：AI NPC 不能直接改游戏世界，只能申请调用被规则系统允许的工具，并把关键交互写入本局记忆。
```

## 阶段 1：Tool Calling / Memory 本地原型

作用：先在本地 Python 里证明“白名单工具”和“本局记忆”两个核心概念。

这一阶段不接网络、不接 Unity、不接真实 MCP。原因是先把最重要的边界跑通：Agent 可以提出工具调用请求，但只有 `get_route_info`、`check_game_state`、`update_quest_hint` 这类白名单工具能执行；像 `open_prison_gate` 这种未授权工具会被拒绝。

对应游戏意义：

- NPC 可以查询路线信息。
- NPC 可以查看受控游戏状态。
- NPC 可以记录玩家问过路线、问过任务等行为。
- NPC 不能直接开门、改血量、传送玩家或乱改任务。

作品集讲法：

```text
我先做了一个本地 Agent 原型，用白名单工具限制 AI 的行动范围。这样 AI 负责理解玩家意图，真正能改变游戏的动作仍然由规则系统控制。
```

## 阶段 2：命令行演示

作用：让本地原型可以一条命令跑出来，方便调试和录屏。

命令行演示不是为了做正式 UI，而是为了让最小功能可见：玩家问路线，程序调用路线工具，返回下方暗道信息，并显示记忆数量。

对应游戏意义：

- 证明 Agent 流程不依赖 Unity 也能单独运行。
- 方便在作品集里展示“输入 -> 工具调用 -> 回复 -> 记忆”的链路。
- 让后续接 FastAPI 或 MCP 前，先有一个稳定基线。

作品集讲法：

```text
命令行版本是我的最小可录制演示，用来证明 Agent 工具调用链路本身能跑通。
```

## 阶段 3：FastAPI 最小接口 `/agent/turn`

作用：把本地 Agent 原型包装成 HTTP 接口，让外部系统可以调用。

这一阶段仍然不接 Unity，也不做长期记忆。它只是证明 Agent Workflow 可以从“只能本地运行的 Python 函数”变成“可以被其他程序请求的服务”。

对应游戏意义：

- 未来 Unity 可以通过 HTTP 调用 Agent。
- 前端、调试工具或录屏脚本也可以调用同一套逻辑。
- `/npc/chat`、`/npc/world_qa` 和 `/agent/turn` 各自独立，避免破坏已经能跑的 AI NPC 和 RAG 接口。

作品集讲法：

```text
FastAPI 阶段把 Agent 原型变成可被外部系统调用的接口，但我没有急着接 Unity，先保证接口边界清楚。
```

## 阶段 4：真实 MCP 最小服务

作用：把工具调用改造成 MCP 风格的 stdio JSON-RPC 服务。

这一阶段证明的重点不是“功能更多”，而是“工具能按协议暴露”。服务支持 `initialize`、`tools/list` 和 `tools/call`，客户端可以看到工具列表，并调用允许的工具。

对应游戏意义：

- 工具能力可以被 MCP 客户端发现。
- 工具调用仍然走白名单校验。
- 未授权工具仍然会被拒绝。
- 游戏状态仍然不交给 AI 自由修改。

作品集讲法：

```text
MCP 阶段把本地工具变成一个协议服务。客户端可以列出工具、调用工具，但工具边界还是由我的 Python 规则系统控制。
```

## 阶段 5：MCP 客户端配置示例

作用：说明真实 MCP 客户端应该怎样启动这个服务。

配置示例记录 `command`、`args` 和 `cwd`。其中 `cwd` 很关键，因为它保证 Python 从当前仓库根目录启动，能够找到 `agent_workflow.mcp_server` 模块。

对应游戏意义：

- 让项目从“只能我自己知道怎么启动”变成“别人能看懂怎么接入”。
- 方便未来接入支持 MCP 的客户端。
- 不直接修改用户本机真实客户端配置，降低风险。

作品集讲法：

```text
我提供了 MCP 客户端配置示例，让这个服务具备可复现的启动方式，而不是只停留在代码里。
```

## 阶段 6：MCP 第二个只读工具 `check_game_state`

作用：证明 MCP 不只能查路线，也可以读取受控游戏状态。

这一阶段只新增只读工具，不暴露 `update_quest_hint`。原因是当前目标是证明“读取状态”已经足够，暂时不让 MCP 客户端修改任务状态。

对应游戏意义：

- AI 可以读取剩余时间。
- AI 可以读取玩家是否有钥匙。
- AI 可以读取当前任务提示。
- AI 仍然不能通过 MCP 直接改任务或开门。

作品集讲法：

```text
第二个 MCP 工具故意选择只读状态查询，用来展示 AI 可以理解局势，但不能越权修改游戏。
```

## 阶段 7：MCP 命令行演示脚本

作用：不用真实 MCP 客户端，也能稳定演示 MCP 工具流程。

演示脚本在进程内模拟三次请求：`tools/list`、`tools/call get_route_info`、`tools/call check_game_state`。它适合录屏，因为不需要先配置外部客户端。

对应游戏意义：

- 展示客户端能看到哪些工具。
- 展示路线工具返回下方暗道信息。
- 展示状态工具返回剩余时间、钥匙和任务提示。
- 展示 MCP 服务当前已经达到最小可讲解程度。

作品集讲法：

```text
我做了一个 MCP 演示脚本，让面试官不用配置客户端，也能看到工具列表、路线查询和游戏状态查询的完整流程。
```

## 阶段 8：阶段作用说明

作用：把前面每个技术阶段翻译成作品集语言。

这一阶段不新增功能，而是回答三个问题：

- 每一步解决什么问题？
- 为什么不提前做更复杂的内容？
- 面试或录屏时应该怎样解释？

对应游戏意义：

- 让技术原型不只是“有代码”，还可以被讲清楚。
- 帮助区分 AI、规则系统、HTTP 接口、MCP 服务和 Unity 的职责。
- 避免继续打磨已经达到 MVP 的 MCP 阶段。

作品集讲法：

```text
我按阶段拆解了 Agent Workflow 的实现路径，这样可以清楚说明每一步的价值，而不是只展示一堆技术名词。
```

## 总结：这一组阶段共同证明什么

这组阶段最终证明了三件事：

1. AI 的行为必须经过工具白名单。
2. 游戏状态必须由可信规则系统控制。
3. 关键玩家行为可以写入本局记忆，为后续 AI NPC 和多 Agent 世界打基础。

对应《囚城营救》的完整表达是：

```text
玩家和 NPC 对话
-> Agent 判断是否需要工具
-> 只能调用允许的工具
-> 工具读取或更新受控游戏字段
-> 玩家关键行为写入本局记忆
-> Unity 未来只执行验证后的结果
```

## 后续不要在本阶段继续做什么

当前阶段已经达到最小可行性，不继续做以下内容：

- 不继续扩展 MCP 工具数量。
- 不接 Unity。
- 不做 LangGraph。
- 不做跨局长期记忆。
- 不把 MCP 改成完整生产级服务。

下一阶段应该进入新的作品集功能，例如 RAG 设定冲突检查器，或回到 Unity 关卡打磨。
