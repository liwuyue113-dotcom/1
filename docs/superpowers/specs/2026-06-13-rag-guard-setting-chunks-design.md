# 胆小守卫 RAG 设定片段优化设计

## 目标

整理现有已确认的胆小守卫设定，使玩家使用同义改写提问时，RAG 更容易检索到直接回答问题的单义片段。

本阶段不创造新剧情，不补充守卫姓名、所属阵营、人质身份或营救原因。

## 当前问题

`docs/rag_setting.md` 中部分段落同时包含多个事实，部分标题范围较宽。BGE 能理解问题语义，但宽泛的游戏设计片段有时会排在更直接的答案前面。

例如：

- “有什么近路能绕过看守？”已经能够命中真实暗道。
- “恐吓那个逃岗士兵，他会怎么指路？”可能先命中守卫身份，而不是虚假路线。
- “如何让躲起来的士兵愿意帮忙？”可能先命中游戏核心玩法，而不是安抚结果。

## 方案

只修改 `docs/rag_setting.md`，按玩家可能提出的问题拆分已确认设定。每个标题下只保留一个主要事实，使标题和正文共同表达答案。

计划使用以下章节：

```text
游戏目标
胆小守卫的身份与位置
安抚胆小守卫的结果
威胁胆小守卫的结果
获得真实路线的条件
获得虚假路线的条件
真实安全路线
虚假危险路线
AI 与规则系统职责
未确认设定
```

## 内容边界

允许整理的事实：

- 胆小守卫因听见砍杀声逃岗，躲在 Level1 深处。
- 胆小守卫认为玩家可能是杀人魔，因此害怕玩家。
- 安抚提高信任并降低恐惧。
- 威胁降低信任并提高恐惧。
- 信任大于等于 50 且信任大于等于恐惧时，守卫提供真实路线。
- 恐惧大于等于信任加 30 时，守卫提供虚假路线。
- 真实路线是往前第二个口子跳下去的暗道，几乎无人镇守。
- 虚假路线是守卫谎称往上走最安全。
- DeepSeek、Python 和 Unity 的职责边界。
- 姬野为什么营救人质仍未确认。

禁止新增的内容：

- 胆小守卫姓名、阵营、职位或过去经历。
- 人质身份、姬野身份或营救原因。
- 城堡势力与剧情结局。
- 尚未实现或未确认的玩法效果。

## 数据流

```text
重写 docs/rag_setting.md
-> Markdown 加载器按标题与段落生成单义片段
-> 使用 BAAI/bge-small-zh-v1.5 重建索引
-> 使用同义改写问题检索 Top 3
-> 检查直接答案是否进入 Top 1 或 Top 3
```

本阶段不修改 `rag/document_loader.py`、Embedding 模型或检索算法。先验证高质量设定文本本身能否解决问题。

## 验收问题

1. `有什么近路能绕过看守？`
   - 真实安全路线片段应进入 Top 1。
2. `恐吓那个逃岗士兵，他会怎么指路？`
   - 威胁结果或虚假危险路线片段应进入 Top 3。
3. `如何让躲起来的士兵愿意帮忙？`
   - 安抚结果或获得真实路线条件片段应进入 Top 3。
4. `不想碰上巡逻兵，该从哪里走？`
   - 真实安全路线片段应进入 Top 3。
5. `主角为什么要营救人质？`
   - 应命中未确认设定，并拒绝编造。

## 验证方式

- 使用项目虚拟环境 `D:\develop\pythons\.venv\Scripts\python.exe` 重建 BGE 索引。
- 运行全部自动化测试。
- 输出五个验收问题的 Top 3 片段与相似度。
- 如果已确认事实未进入规定排名，再评估是否需要修改加载器或检索策略。

## 本阶段不做

- 不新增世界观设定。
- 不修改切片算法。
- 不调整相似度阈值。
- 不接入 DeepSeek 在线回答。
- 不制作新的 UI。
