# 《囚城营救》RAG 游戏设定知识库

## 当前目标

第一版 RAG 用于回答《囚城营救》的游戏设定问题，并显示实际检索来源。

核心流程：

```text
精选设定 Markdown
-> 按标题和段落切分
-> 组合片段标题与正文作为索引文本
-> BAAI/bge-small-zh-v1.5 生成中文语义向量
-> NumPy 余弦相似度检索
-> 根据证据回答
-> 显示来源文件
```

## 当前索引范围

- `docs/rag_setting.md`

`docs/rag_setting.md` 只保存已确认、适合检索的单义设定。完整策划案 `docs/game_design.md`、开发进度、技术决策、交接记录和备选玩法不进入设定问答索引。

## 已完成功能

- Markdown 标题与段落切分。
- 文本片段保存来源与标题。
- 本地向量索引构建与持久化。
- NumPy 余弦相似度 Top 3 检索。
- DeepSeek 约束回答 Prompt。
- DeepSeek 或 API Key 不可用时显示最相关原文。
- 依据不足或设定尚未确认时拒绝编造。
- 命令行连续问答。
- 已安装并启用正式中文 Embedding 模型 `BAAI/bge-small-zh-v1.5`。
- 构建索引时使用“标题：正文”生成向量，检索展示仍保留原始正文。

## 验收结果

已于 2026-06-12 完成第一版验收：

- `胆小守卫提供的真实路线在哪里？`
  - 正确检索往前第二个口子跳下去的暗道。
- `威胁胆小守卫可能得到什么情报？`
  - 正确检索往上走最安全的虚假情报。
- `玩家为什么要营救人质？`
  - 在核心剧情尚未确认时，正确回答当前知识库没有足够信息。

自动测试：31 项通过。

正式中文 Embedding 模型已于 2026-06-12 完成验收：

- BGE 索引包含 26 个设定片段，向量维度为 512。
- 同义改写问题“有什么近路能绕过看守？”能够正确检索到第二个口子下方的暗道。
- 相比内置文本向量兜底，BGE 能理解“近路”“绕过看守”与“安全暗道”的语义关系。

## 当前限制

- 必须在已安装 `sentence-transformers` 的 Python 虚拟环境中构建索引和运行问答，避免 BGE 索引与兜底查询向量维度不一致。
- 宽泛问题可能同时匹配多个相关设定。例如“如何让躲起来的士兵愿意帮忙？”的 Top 1 不是安抚结果，但真实路线条件已进入 Top 3。
- DeepSeek 回答可能包含 Markdown 标记，例如 `**加粗**`；当前命令行会直接显示标记，不影响答案事实。

## 单义设定片段验收

已于 2026-06-13 将胆小守卫设定按玩家问题拆分为单义片段，让标题参与 Embedding，并将索引范围限制为专用已确认设定文档。五个同义改写问题的 BGE Top 3 验收全部通过。

通过：

- `有什么近路能绕过看守？`：`真实安全路线` 为 Top 1。
- `恐吓那个逃岗士兵，他会怎么指路？`：`虚假危险路线` 为 Top 2。
- `如何让躲起来的士兵愿意帮忙？`：`获得真实路线的条件` 为 Top 3。
- `不想碰上巡逻兵，该从哪里走？`：`真实安全路线` 为 Top 1。
- `主角为什么要营救人质？`：在核心剧情尚未确认时，`未确认设定` 为 Top 1，并回答当前知识库没有足够信息。

结论：专用设定白名单移除了完整策划案中的通用玩法干扰，当前最低可用语义检索阶段已经完成。宽泛问题的 Top 1 排名仍有优化空间，但不阻塞进入下一阶段。

## DeepSeek 真实回答验收

已于 2026-06-15 使用在线 DeepSeek 完成四题真实回答验收：

- `真正安全的路线是什么？`
  - 正确回答往前第二个口子跳下去的暗道，几乎无人镇守。
- `恐吓胆小守卫后，他会怎么指路？`
  - 正确回答威胁会降低信任、提高恐惧，并可能获得虚假路线。
- `怎样才能让胆小守卫提供真实路线？`
  - 正确回答信任大于等于 50，并且信任大于等于恐惧。
- `姬野为什么要营救人质？`
  - 在核心剧情尚未确认时，正确回答当前知识库没有足够信息。

四题均显示来源 `docs/rag_setting.md`。验收期间没有出现“未调用 DeepSeek”或“DeepSeek 调用失败”。

## 核心剧情设定验收

已于 2026-06-15 确认并写入《九州缥缈录》原著人物与世界观下的原创同人支线：

- 姬野为了营救昔日战友吕归尘，独自潜入离国边境城堡。
- 吕归尘在战争中被离国军俘获，并被关押审问。
- 离国军将在黎明前把吕归尘转移至主力军营。
- 姬野找到吕归尘后关卡完成；倒计时结束前未找到则失败。
- 当前版本不包含找到吕归尘后的护送与逃离玩法。

使用 BGE 重建 17 个专用设定片段后，五个剧情问题的 Top 1 均命中正确片段。

## 三条潜入路线设定验收

已确认当前可用道路为三条主路：

- 上路：玩家可以直接选择，路线较长，敌人数量一般。
- 中路：玩家可以直接选择，长度一般，敌人数量略多。
- 下方暗道路线：通过胆小守卫真实情报发现，长度正常，敌人最少，往前第二个口子跳下去有暗道。

使用 BGE 重建 21 个专用设定片段后，上路、中路和下方暗道路线问题的 Top 3 均命中对应片段。
目前不再记录“城堡外侧、侧门、正门上层、牢房区五区域”作为已确认设定，避免 RAG 回答出实际 Unity 地图中不存在的道路。

## 三条路线提示方式验收

已确认三条路线的最低可用提示方式：

- 上路：入口看起来更远、更绕，可用较长走廊、拐弯或远处火把表现。
- 中路：入口附近能看到更多守卫或更密集的巡逻点。
- 下方暗道路线：默认不明显，需要胆小守卫真实情报提示“往前第二个口子跳下去”。

使用 BGE 重建 24 个专用设定片段后，上路提示、中路提示和下方暗道提示三个问题的 Top 3 均命中对应片段。

## 三条路线敌人站位验收

已确认三条路线的最低可用敌人站位：

- 上路：开始很分散，主要是一个一个的守卫；后面会有连续几个守卫站岗。
- 中路：敌人分布很多，有的地方有好几个守卫站岗。
- 下方暗道路线：只有暗道内部有一个站岗的守卫。

使用 BGE 重建 27 个专用设定片段后，上路、中路和下方暗道敌人站位三个问题的 Top 3 均命中对应片段。

## 三条路线巡逻方式验收

已确认三条路线的最低可用巡逻方式：

- 上路：前半段守卫分散站岗或短距离巡逻，后半段连续几个守卫偏固定站岗，形成压力区。
- 中路：守卫更密集，部分位置有多个守卫同时站岗，玩家更容易战斗或等待空隙。
- 下方暗道路线：暗道内部只有一个站岗守卫，不做复杂巡逻，主要挑战是找到暗道入口。

使用 BGE 重建 30 个专用设定片段后，上路、中路和下方暗道巡逻方式三个问题的 Top 3 均命中对应片段。

## 设定冲突检查器最小原型

已于 2026-06-24 完成第一版 RAG 设定冲突检查器。

作品集演示说明见 `docs/rag_conflict_checker_demo.md`。

当前作用：

- 输入一段候选新增设定。
- 复用现有 RAG 索引检索最相关的已确认设定片段。
- 输出“可能冲突 / 暂未发现明显冲突 / 证据不足”。
- 显示冲突原因、来源文档和相关设定片段。

当前第一版只使用明确规则判断明显冲突，不调用大模型，也不自动修改设定。例如：

```text
新增设定：下方暗道内部有多个守卫来回巡逻。
```

会与已确认设定“下方暗道内部只有一个站岗守卫”形成可能冲突。

运行命令：

```powershell
D:\develop\pythons\.venv\Scripts\python.exe -m rag.check_conflict "新增设定：下方暗道内部有多个守卫来回巡逻。"
```

当前边界：

- 不接 Unity。
- 不调用 DeepSeek。
- 不自动改写 `docs/rag_setting.md`。
- 不替代人工策划判断，只作为新增设定前的提醒工具。
