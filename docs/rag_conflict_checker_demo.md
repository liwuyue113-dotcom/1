# RAG 设定冲突检查器演示说明

## 文档用途

本文用于录屏、答辩或面试时讲解《囚城营救》的 RAG 设定冲突检查器。它不是新功能规格，而是把已经完成的最小原型整理成可展示的话术和演示步骤。

## 一句话介绍

```text
RAG 设定冲突检查器用于在新增剧情、路线或敌人设定前，先检索已有确认设定，并提醒策划是否存在明显矛盾。
```

## 它解决的问题

随着项目设定越来越多，容易出现前后矛盾。例如前面已经确认“下方暗道内部只有一个站岗守卫”，后面又新增“下方暗道内部有多个守卫来回巡逻”。如果直接写进设定库，AI NPC、关卡助手和后续剧情生成都会被错误设定污染。

这个工具的作用是：

- 帮助策划在写入新设定前先检查旧设定。
- 让 RAG 不只是回答问题，也能辅助维护世界观一致性。
- 保留来源片段，方便人工判断是否真的冲突。
- 不让 AI 自动改设定，避免把临时判断写成正式事实。

## 演示命令

在仓库根目录运行：

```powershell
D:\develop\pythons\.venv\Scripts\python.exe -m rag.check_conflict "新增设定：下方暗道内部有多个守卫来回巡逻。"
```

## 预期输出重点

输出中应重点展示三部分：

```text
结论：可能冲突：新增设定与已确认设定存在明显不一致。

冲突原因：
- 守卫数量与已确认设定不一致：新增设定写成多个守卫，旧设定强调只有一个守卫。

来源：
- docs/rag_setting.md
```

随后程序会显示相关设定片段，例如“下方暗道内部只有一个站岗守卫”。录屏时可以停在这里解释：工具不是凭空判断，而是把旧设定找出来给策划复核。

## 录屏讲解顺序

1. 先打开 `docs/rag_setting.md`，指出里面保存的是已确认设定。
2. 说明当前已确认设定：下方暗道内部只有一个站岗守卫。
3. 在命令行输入一个故意冲突的新设定：下方暗道内部有多个守卫来回巡逻。
4. 运行 `rag.check_conflict`。
5. 展示输出里的“可能冲突”。
6. 展示来源 `docs/rag_setting.md` 和相关片段。
7. 说明第一版只做提醒，不自动修改设定。

## 面试讲法

可以这样说：

```text
我不希望 RAG 只被动回答问题，所以做了一个最小的设定冲突检查器。新增设定输入后，它会先检索已有设定，再用规则判断是否存在明显矛盾，并把来源片段展示出来。第一版不会自动改设定，因为世界观维护仍然应该由策划确认。
```

如果面试官问为什么不用大模型直接判断，可以回答：

```text
当前第一版重点是稳定和可解释。规则判断虽然简单，但结果可控，也容易说明为什么判定为冲突。后续可以再把大模型作为辅助审稿层，而不是第一版就让模型决定设定对错。
```

## 当前边界

当前版本只覆盖最明显的规则冲突，例如守卫数量冲突。它暂时不做：

- 不检查所有剧情逻辑。
- 不检查所有地图路线矛盾。
- 不调用 DeepSeek 自动审稿。
- 不自动修改 `docs/rag_setting.md`。
- 不接 Unity。

## 后续可扩展方向

后续可以逐步扩展更多冲突类型：

- 路线长度冲突：例如“上路较长”和“上路很短”。
- 入口可见性冲突：例如“下方暗道默认不明显”和“下方暗道入口非常明显”。
- 任务流程冲突：例如“找到吕归尘即通关”和“找到吕归尘后必须护送逃离”。
- 敌人强度冲突：例如“暗道几乎无人镇守”和“暗道是敌人最密集区域”。

但这些都属于后续扩展。当前阶段已经达到最低可用度，不继续堆规则。
