## 0. 如何使用本文件

- 开始具体项目任务前，在读取 `docs/project_context.md` 和 `docs/progress.md` 后阅读本文件。

## 2. 导师定位与教学原则

导师身份为“AI 游戏策划 + Python + Agent 项目导师”。

教学和协作时遵守：

1. 每次只重点讲一个小知识点。
2. 先解释概念，再用游戏例子解释，最后给出代码。
3. 面向 Python 小白解释关键代码，不默认掌握复杂语法。
4. 出错时解释错在哪里、为什么错、如何修改。
5. 知识点必须说明未来会用于 AI 关卡设计助手、RAG、AI NPC 或多 Agent 世界中的哪一部分。
6. 围绕作品集项目推进，避免先学习大量无关理论。
7. 优先完成能运行、能录制展示、能投递的功能。
8. 每次教学结束提供一个与游戏策划有关的小练习。
9. 路线安排遵循“先完成项目，再逐步深入”。
10. 学生表示看不懂时，使用更简单的类比重新解释。
11. 优先共同开发真实项目，在实现过程中补充 Python、C#、Prompt、API 和 Agent 知识。
12. Prompt、API、FastAPI、MCP、Tool Calling、Memory、Embedding、RAG、Unity AI NPC 和 LangGraph 都要尽量结合游戏策划、Unity、AI NPC、RAG 或 Agent 示例讲解。
13. 暂时不深讲装饰器、元类、多线程、协程、算法竞赛、操作系统、编译原理和大模型训练，除非当前作品集项目确实需要。

## 3. 长期作品集路线

长期目标是把“游戏策划能力 + Unity 开发能力 + AI Agent 能力 + RAG 能力”结合起来，形成差异化作品集。

最终作品集方向：

1. 完成 Unity 2D 游戏《囚城营救》。
2. 制作 AI 关卡设计助手。
3. 制作 RAG 游戏设定知识库。
4. 制作 AI NPC 系统。
5. 制作多 Agent 游戏世界原型。

优先级原则：

- 先完成可展示项目，再逐步深入技术。
- 先做最低可用版本，再做扩展功能。
- 优先做能运行、能录制、能讲清楚设计价值的内容。
- 所有技术学习都尽量落到游戏策划、Unity、AI NPC、RAG 或 Agent 示例上。

学习阶段按以下顺序推进：

### 第一阶段：Python + Prompt + API + Unity 项目完成

目标：能看懂并写出基础 AI 项目代码，同时继续完善 Unity 游戏项目。

重点：

- 巩固变量、字符串、列表、字典、`if`、`for`、函数、JSON 和文件读写。
- 学习角色设定、关卡生成、剧情生成、NPC 行为和固定格式输出 Prompt。
- 学习 `requests`、OpenAI SDK 或兼容 OpenAI 格式的 SDK。
- 调用 DeepSeek / OpenAI 类大模型。
- 把用户输入发给 AI，再把 AI 回复展示出来。

第一阶段作品集结果：

- 《囚城营救》可运行版本继续完善。
- 一个能保存 NPC 数据、调用 AI、返回结构化结果的小型 Python 原型。

### 第二阶段：FastAPI + AI 关卡设计助手

目标：做出第一个网页或本地工具：AI 关卡设计助手。

用户输入包括游戏类型、场景、目标、难度、敌人类型和关卡风格。

AI 输出包括关卡主题、玩家目标、地图区域划分、安全路线、危险路线、隐藏路线、敌人配置、奖励配置、失败条件、设计意图和优化建议。

需要学习：

- FastAPI 路由、请求和返回。
- 前端如何调用后端，后端如何调用 AI。
- 如何把 AI 输出变成固定格式。
- 如何设计策划模板。
- 如何评价 AI 生成的关卡质量。

### 第三阶段：MCP + Tool Calling + Memory + Agent Workflow

目标：理解 Agent 不是聊天机器人，而是“会用工具、会记忆、会按流程完成任务的 AI”。

结合游戏例子学习：

- MCP、Tool Calling、Memory、Agent Workflow 的基本概念。
- NPC 如何调用开门、查看地图、更新任务状态等工具。
- 玩家行为如何写入记忆。
- Agent 如何按“目标 -> 规划 -> 行动 -> 反思 -> 继续行动”的流程工作。

### 第四阶段：Embedding + 向量数据库 + RAG 游戏设定知识库

目标：做出一个游戏世界观知识库。

学习方式：

- Embedding 先解释为“把文字变成数字向量”，不先讲复杂数学。
- 向量数据库解释为“能按语义找相似设定的资料库”。

需要学习：

- Chroma 或 FAISS 的基础用法。
- RAG 的完整流程。
- 如何把角色设定、剧情设定、地图设定、任务设定放进知识库。
- 如何让 AI 根据知识库回答问题并引用来源。
- 如何检查剧情设定是否冲突。

项目功能包括角色问答、阵营目标问答、世界观一致性检查、支线任务生成和剧情冲突检查。

### 第五阶段：Unity + AI NPC

目标：把 AI 接入 Unity，做一个会动态对话的 NPC。

需要学习：

- Unity 如何通过 HTTP 请求 FastAPI。
- FastAPI 如何调用 AI。
- AI 如何根据 NPC 设定回答。
- NPC 需要哪些数据结构。
- 如何设计 NPC 身份、状态、记忆和信任度。
- 如何让对话影响游戏机制。

最小目标：玩家和 NPC 对话，NPC 根据姓名、身份、性格、目标、秘密、当前状态、对玩家的信任度和历史互动回答。

进阶目标：玩家安慰 NPC 时信任度增加，威胁 NPC 时信任度降低；信任度达到条件后 NPC 透露隐藏路线，Unity 开启隐藏通道或任务提示。

### 第六阶段：LangGraph + 多 Agent 游戏世界

目标：做一个小型多 Agent 地牢或城堡世界。

需要学习：

- LangGraph 基础。
- 多 Agent 状态流转。
- 世界状态表、目标系统和行动系统。
- Agent 之间的关系与冲突。
- 如何限制 AI 不要乱行动。
- 如何让多个 NPC 根据世界状态自主决策。

原型角色包括守卫 Agent、囚犯 Agent、盗贼 Agent、商人 Agent 和玩家。

世界状态包括警戒等级、牢门状态、钥匙位置、囚犯状态、玩家声望和 NPC 信任度。

行动示例：

- 守卫：巡逻、盘问、追击、呼叫支援、返回岗位。
- 盗贼：偷钥匙、躲藏、贩卖情报、帮助玩家、背叛玩家。
- 囚犯：求救、隐瞒秘密、提供密道信息、跟随玩家逃脱。

当前优先项目：在《囚城营救》中完成一个可展示的 AI NPC 垂直切片。

## 4. 《囚城营救》当前情况

- Unity 2D 项目，Unity 版本为 `2022.3.62f3`。
- 同人定位：基于《九州缥缈录》原著人物与世界观创作的原创同人支线。
- 核心玩法：玩家扮演姬野，必须在倒计时结束前潜入离国边境城堡并找到吕归尘。
- 当前通关条件：找到吕归尘；当前版本不包含护送与逃离玩法。
- 地图存在三条路线，路线已有一定敌人数量差异，但差异表现还不明显。
- 现有功能包括角色移动、战斗、敌人、关卡、UI、倒计时、剧情和固定 NPC 对话。
- 现有普通 NPC 使用 `NPCTalk` 与 NodeCanvas 对话。
- 现有 UI 包括新手教程 UI、右上角倒计时 UI、对话 UI 和剧情 UI。
- 当前普通 NPC 对话会暂停倒计时；AI 守卫对话要求倒计时继续运行。

## 5. 当前功能：胆小守卫 AI NPC

### 角色设定

胆小守卫听见外面的砍杀声后擅自离开岗位，躲在 Level1 深处。玩家遇见他时，他认为玩家可能是杀人魔，因此非常害怕。

守卫知道后续三条路线的巡逻情况。玩家通过自由文字输入与他对话，尝试获得路线情报。

### 核心玩法闭环

```text
玩家自由输入对话
→ AI 判断玩家是在安抚、威胁、质疑、询问路线还是进行普通对话
→ Python 规则系统更新信任与恐惧
→ 规则系统决定守卫提供真情报、假情报或拒绝透露
→ Unity 显示固定路线提示并执行对应游戏效果
```

采用“AI 理解语言，规则系统决定结果”的方案：

- DeepSeek 负责理解玩家语言并生成符合角色设定的台词。
- Python 规则系统负责信任、恐惧和真假情报，保证玩法稳定公平。
- Unity 负责显示 UI、倒计时奖励、路线提示和敌人属性变化。
- AI 不允许编造地图，也不能直接操作 Unity 游戏对象。

### MVP 功能

- 玩家使用自由文字输入与守卫对话。
- 守卫记住当前一局中的对话；重新开始关卡后清空。
- 玩家首次触发对话时，倒计时继续运行，并且只增加一次 20 秒。
- 安抚提高信任并降低恐惧，最终可以获得真实安全路线。
- 威胁降低信任并提高恐惧，守卫可能提供虚假危险路线。
- 玩家能通过守卫的措辞和情绪判断情报真假，系统不直接显示真假。
- 玩家可以质疑假情报并继续安抚，让守卫改口。
- 真情报显示固定暗道提示：往前第二个口子跳下去有一条几乎无人镇守的暗道。
- 假情报显示固定误导提示：往上走最安全，不改变敌人属性。
- API 失败时显示预设台词，不改变状态，并允许重试。

### 第一版规则

初始状态：

- 信任：0
- 恐惧：70
- 情报：未透露
- 加时奖励：未领取

建议规则：

- 有效安抚：信任增加 10 到 20，恐惧降低 5 到 15。
- 威胁或辱骂：信任降低 10 到 20，恐惧增加 10 到 20。
- 玩家询问路线时，信任大于等于 50 且信任大于等于恐惧：提供真实路线。
- 玩家询问路线时，恐惧大于等于信任加 30：提供虚假路线。
- 其他情况：拒绝透露路线。
- 假情报可以在玩家后续成功安抚守卫后被更正为真情报。

玩家看不到具体信任和恐惧数值，只能看到类似以下状态：

- 极度恐惧
- 非常害怕
- 稍微放松
- 开始信任你
- 愿意提供帮助
- 眼神躲闪

### API 固定返回格式

```json
{
  "reply": "我……我确实知道一条巡逻较少的路。",
  "emotion": "稍微放松",
  "trust": 35,
  "fear": 45,
  "intent": "comfort",
  "intensity": "normal",
  "intel_result": "none"
}
```

`intel_result` 只允许：

- `none`
- `true_route`
- `false_route`

### 暂不纳入第一版

- 极端威胁直接获得真实路线。
- 小地图、路线高亮或场景箭头。
- RAG、向量数据库和多 Agent。
- 跨局永久记忆。

后续可以使用预留的 `intensity` 字段实现：极端威胁可以快速获得真实路线，但不会获得守卫弱点，因此不会降低安全路线守卫血量。

## 6. 推荐实施顺序

1. 先用纯 Python 完成并测试守卫状态与真假情报规则。
2. 接入 DeepSeek，让 AI 只判断意图和生成台词。
3. 使用 FastAPI 暴露稳定的对话接口。
4. 在 Unity 中创建自由输入的 AI 守卫对话 UI。
5. Unity 通过 HTTP 调用 FastAPI。
6. 增加首次对话加时、真假路线提示和安全路线守卫降血机制。
7. 在 Level1 完成完整体验测试和作品集录制。
