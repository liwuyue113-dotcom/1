# 《囚城营救》作品集项目总规则

本文件是仓库内 AI 协作的**主规则文件**，用于规定所有项目任务都必须遵守的工作方式。

详细项目背景、教学方式与当前 AI NPC 功能方案记录在 `agent.md`。执行具体任务时，必须按下方顺序读取资料。

## 1. 信息优先级

信息发生冲突时，按以下优先级判断：

1. 用户在当前对话中明确提出的最新要求
2. 当前代码、场景和实际运行结果
3. 本文件 `AGENTS.md` 中的仓库总规则
4. `agent.md` 中的详细项目协作说明与当前功能方案
5. `docs/` 中的项目知识库
6. 未验证的聊天历史、猜测或旧方案

不要因为文档已经存在，就默认其中所有内容仍然正确。与任务有关的信息应结合代码和运行结果验证。

## 2. 开始任务前

处理仓库任务前，按顺序阅读：

1. `docs/project_context.md`
2. `docs/progress.md`
3. `agent.md`
4. 与当前任务相关的设计文档

相关设计文档：

- 游戏策划与关卡设计：`docs/game_design.md`
- 技术方案与开发决策：`docs/tech_decision.md`
- AI NPC 系统：`docs/ai_npc.md`
- 长期学习路线与作品集目标：`agent.md` 第 3 节
- 编码与项目管理规范：`docs/coding_rules.md`

只需深入阅读与当前任务有关的部分，避免无目的加载全部资料。

## 3. 项目总目标

本仓库用于长期维护个人作品集项目，包含：

- Unity 2D 潜入营救游戏《囚城营救》
- Python + Prompt + API 基础学习
- AI NPC 系统
- RAG 游戏设定知识库
- AI 关卡设计助手
- MCP / Tool Calling / Memory / Agent Workflow
- LangGraph 与多 Agent 游戏世界原型

优先完成能够运行、录制展示和清楚讲解的功能。避免为了使用新技术而加入与作品集目标无关的复杂设计。

当前第一优先级：完成《囚城营救》的“胆小守卫 AI NPC”垂直切片。具体玩法和实施顺序见 `agent.md`。

## 5. 开发原则

- AI 负责理解语言和生成符合角色设定的台词。
- Python 规则系统负责信任、恐惧、真假情报等稳定玩法规则。
- Unity 负责可信游戏状态、UI 和游戏效果。
- AI 返回的行为建议必须经过规则验证后才能改变游戏状态。
- 不在代码、文档或提交中保存 API Key、密码或其他秘密。
- 未知信息应明确标记为待确认，不得凭空补全。

## 6. 教学与协作原则

- 采用项目驱动方式，共同开发真实功能，在需要时讲解知识点。
- 每次重点讲一个小知识点。
- 先解释概念，再结合游戏例子，最后展示适合初学者的代码。
- 出错时解释错在哪里、为什么错、如何修改。
- 说明知识点未来会用于 AI NPC、RAG、关卡助手或多 Agent 世界的哪一部分。
- 优先使用直观方案，不为了显得高级而引入复杂语法或架构。
- 教学任务结束时，提供一个与游戏策划或当前功能有关的小练习。
- Prompt、API、FastAPI、MCP、Tool Calling、Memory、Embedding、RAG、Unity AI NPC 和 LangGraph 都必须结合游戏策划与作品集项目讲解。
- 暂时不深讲装饰器、元类、多线程、协程、算法竞赛、操作系统、编译原理和大模型训练，除非当前项目确实需要。

更详细的学生背景和教学要求见 `agent.md`。

### 阶段式项目教学规则

从 2026-06-08 起，项目必须按阶段推进：

1. 每个阶段只实现当前阶段的最低可用版本，不提前做高级优化。
2. 用户每次反馈运行结果后，必须先进行阶段验收：
   - 当前阶段目标是否完成？
   - 是否存在阻塞下一阶段的 bug？
   - 如果没有阻塞问题，必须进入下一阶段。
3. 一旦某阶段达到最小可行性，必须停止继续补充、打磨或扩写该阶段内容，并开始下一阶段。
4. 禁止在没有必要的情况下反复优化同一段代码或同一类文档设定。
5. 只有代码无法运行、逻辑明显错误，或问题会阻塞下一阶段时，才修改当前代码或继续补当前阶段文档。
6. 每次教学回复必须明确包含：
   - 当前阶段
   - 本阶段目标
   - 是否完成
   - 下一步要做什么
   - 本次只需要用户修改哪些文件
7. 不因为未来可扩展性提前升级架构，先完成能运行的最低可用版本。
8. 如果在同一段代码或同一类文档设定停留超过 2 轮，必须主动提醒：
   - “当前阶段已完成，建议进入下一阶段。”
9. 项目路线与阶段验收标准见 `docs/staged_roadmap.md`。

## 7. 完成任务后

完成项目功能、修复或重要设计讨论后，应根据实际变化更新知识库：

1. 更新 `docs/progress.md` 中的已完成、开发中和待开发状态。
2. 更新与任务直接相关的设计文档。
3. 将重要技术选择、备选方案和原因记录到 `docs/tech_decision.md`。
4. 项目整体方向或开发者长期情况变化时，更新 `docs/project_context.md`。
5. 新增可复用开发规则时，更新 `docs/coding_rules.md`。
6. 当前 AI NPC 角色、规则或实施顺序发生变化时，更新 `agent.md` 和 `docs/ai_npc.md`。

只记录已确认、值得长期保留的信息。不要保存普通闲聊、临时猜测或已经被否定的方案。

## 8. 验证与交付

- 完成功能后运行适合该功能的验证。
- 最终回复说明完成内容、验证结果和仍存在的风险。
- 不把重要项目信息只保留在聊天记录中。
- 修改文档时保留已有有效信息，避免无关重写。
